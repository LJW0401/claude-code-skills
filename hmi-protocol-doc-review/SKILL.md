# hmi-protocol-doc-review

针对嵌入式 / 工控类项目常见的两份配套文档做一致性审核：

- **通信协议文档**：定义帧结构、功能码、字段含义、错误码
- **上位机界面设计文档**：定义 UI 布局、按钮逻辑、输入项、显示项

二者通常各自独立维护，最容易出现**协议改了但 UI 没同步**、**白板时序图功能码画错**、**字段单位缺失**、**数据长度算错**之类的问题。本 skill 把审核流程和高频坑点固化下来。

## 工作流

1. **确认资源** —— 问用户拿到两份文档的位置（飞书 docx token / URL / 本地路径），以及是否嵌有白板（流程图、布局图）。白板内容必须单独读图，不能只看 docx 文字。
2. **全文拉取** ——
   - 飞书 docx：`lark-cli docs +fetch --doc <token> --format pretty`
   - 飞书白板：`lark-cli docs +media-download --type whiteboard --token <token> --output ./xxx.png --overwrite`，然后用 Read 工具看图
   - 本地：用 Read 直接读
3. **逐项对账**（见下方检查清单），按严重程度 🔴/🟡/🟢 分级。
4. **不要默认改文档** —— 列出问题让用户拍板；用户说"修"再用 `docs +update --mode replace_range --selection-by-title` 改。
5. **修改后必须重新 fetch 验证**，不要相信 `success: true` 就完事。

## 检查清单

### 🔴 严重（直接影响协议正确性 / 解析错位）

#### 1. DATA 长度对账
协议帧表里的"数据长度"字段必须等于数据域表中所有字段字节数之和。
- 公式：u8=1, u16=2, u32=4, float=4, double=8
- 每次数据域增删字段都要重算 DATA 长度
- **最高频踩坑点**：开发者在数据域加 / 删一个字段，忘了同步改 DATA 长度 → 下位机/上位机解析错位、CRC 位置错、整条帧报废

#### 2. 功能码一致性（白板 / 时序图 vs 协议章节）
如果文档里有用白板 / 时序图画通信流程，每个流程标注的功能码必须和协议章节的功能码定义一致。
- 常见踩坑：画图时复制粘贴上一个流程框忘改功能码（例如"停止反馈"画成了"参数更新"的功能码）
- 不仅看功能码数字，**还要看流程框里的文字说明**是否对应（例如功能码改了但说明文字还是旧的）

#### 3. UI 输入项数量 = 协议字段数量
下行参数帧（上位机 → 下位机）有几个字段，UI 上就必须有几个输入框 / 控件。
- 缺一个 = 那个字节没数据来源，要么硬编、要么发非法值
- 多一个 = UI 上有但协议没承载，用户改了没生效

#### 4. 上行反馈字段 = UI 显示项
传感器 / 状态上报帧的每个字段都要在 UI 上有显示位置（数值卡片 / 图表 / 状态灯 / 按钮）。否则数据收了没用。

#### 5. 协议定义了但 UI 没触发入口的"死帧"
检查每个功能码在 UI 文档里是否能找到触发路径（按钮、自动时机、定时心跳）。如果一个协议帧定义了但 UI 不会发，就是死代码。

#### 6. 位图 / 枚举字段的 UI 展示方式
错误码、状态字这类**位图**字段不能用单值显示。
- 多 bit 可同时置位 → UI 必须支持同时展示多个故障
- bit 含义对照表应在 UI 弹窗中直接列出，而不是"参考协议文档"跨文档跳转

### 🟡 设计层面

#### 7. ACK / 重试策略一致
- 协议文档应明确"几次无应答 = 异常"
- UI 文档的"ACK 异常弹窗"逻辑必须和协议侧的重试次数对齐
- 不一致 → 偶发丢包会被频繁弹窗骚扰

#### 8. 跨文档命名同步
按钮名 ↔ UI 文档描述 ↔ 协议文档功能描述 ↔ 白板按钮标签。
- 改了其中一处，其他地方要同步
- 命名要避免误导（例如"清空错误码"按钮可能让用户以为清当前故障，实际只是清计数器）

#### 9. 单位标注完整性
每个数值字段（尤其是反馈）都要标单位：
- 转速：rps / rpm / Hz —— 区分清楚
- 功率：W（规范大写）
- 电压：V / mV
- 电流：A / mA / 0.1A
- 温度：°C / 0.1 °C
- 压力：MPa / kPa
- 时间：ms / μs
- 占空比 / 比例：% / 0.1%
- 重量 / 位置原始值：必须给量程或换算系数
- 时间戳：注意 u32 约 49.7 天回卷、u16 约 65 秒回卷

#### 10. 字节序约定
小端 / 大端必须在协议文档明确写。即便约定了，也要检查 UI 解析代码假设是否一致。

#### 11. 校验算法明确
CRC8 / 16 / 32 有多种 polynomial / init / reflect / xorout 组合，必须写明（例如 "CRC-8/MAXIM"、"CRC-16/MODBUS"）。

#### 12. 量程 / 范围边界
协议字段的合法范围（如 `[30,90]`）要在：
- 协议文档定义
- UI 输入做范围校验
- UI 文档里能查到（直接列或明确引用协议）

#### 13. 启停 vs 参数设定的联动语义
当起停命令和参数设定拆成两个帧时，需要明确：
- 启动顺序：先设参数还是先启动？
- 运行中改参数是否立即生效？
- 停机后设定值是否保留？

#### 14. 长时间运行的回卷 / 累积误差
- u32 时间戳：49.7 天回卷
- 累积计数器：溢出后图表会跳变，需 wrap-around 处理

#### 15. 状态机切换的边界态
- 数据反馈态 ↔ 不反馈态切换时，显示数据如何处理（保留 / 清空 / 置灰）？
- 连接断开 / 恢复时，状态显示如何重置？

### 🟢 小问题

- **错别字 / 重字**：常踩"的的"、"在不"、"是是"
- **空标题占位**：标题后无内容，补"等待补充"或删除
- **括号嵌套**：例如 "脉冲（pulses）"、"开度（%开度）"
- **大小写**：物理单位 W / V / A 必须大写
- **链接 / 引用失效**：跨文档引用对方章节，章节名改了 / 删了

## 输出格式

```markdown
## 🔴 严重
1. **<问题>** — <位置：章节标题 / 行号 / 白板区块> — <建议修法>

## 🟡 设计层面
...

## 🟢 小问题
...

## ✅ 已对齐
- ...
```

每条问题必须给出具体位置（章节标题 / 行号 / 白板区块），避免泛泛而谈。

## 修文档时的注意事项

- **整段重写最稳**：`lark-cli docs +update --mode replace_range --selection-by-title "<完整标题>"`
- **修改前先 fetch 底稿**：避免覆盖用户的其他改动
- **白板内容无法用 lark-doc 编辑**：只能让用户在飞书里手动改，或走 lark-whiteboard 全量覆盖（破坏性）
- **v1 API 有 `[deprecated]` 警告但仍可用**；如果用 v2，用 `--api-version v2 --command str_replace --pattern <regex>`
- **传文件路径要用相对路径**：`@./xxx.md`，不能 `@/tmp/xxx.md`（lark-cli 限制 cwd 内相对路径）
- **修改后必须重新 fetch 验证**

## 常见踩坑（来自历史经验）

1. **数据域字段增删时忘改 DATA 长度** —— 最高频，每次必查
2. **白板复制流程框忘改功能码** —— 高频
3. **拆字段后两个字段对齐错位** —— 比如把单字段 u16 拆成两个 u16 后，DATA 长度只加了 2 没加 4
4. **协议改了但 UI 文档没同步** —— 按钮名、字段数、范围都可能漂
5. **白板里的描述文字和功能码不匹配** —— 功能码改对了但说明文字还是旧的
6. **"清空 X" 类按钮的语义** —— 究竟清的是当前值还是计数器要写清楚
7. **位图字段当数值显示** —— UI 把 u16 位图当成枚举或十进制数显示
8. **死帧** —— 协议定义了某个帧但 UI 没任何触发入口
