# hmi-protocol-doc-review

针对嵌入式 / 工控类项目常见的两份配套文档做质量审核：

- **通信协议文档**：定义帧结构、功能码、字段含义、错误码
- **上位机界面设计文档**：定义 UI 布局、按钮逻辑、输入项、显示项

二者通常各自独立维护，最容易出现**协议改了但 UI 没同步**、**白板时序图功能码画错**、**字段单位缺失**、**数据长度算错**之类的问题。

## 工作模式（先判定再开干）

进入 skill 第一步是确认本次审核的对象是哪一份文档，**不要同时摊开两份混着审**。

### 模式 A：单审「通信协议文档」

**触发**：用户说「检查通信文档」「审协议」「看下协议有没有问题」，或只给出协议文档的 URL/路径。

**范围**：只看协议文档自身，不主动去找配套的 UI 文档。即便用户随口提了 UI，也明确告知"本轮只审协议自洽性"。

**只跑这些检查项**：
- 🔴 协议自洽：DATA 长度对账、白板 vs 协议章节功能码一致、位图/枚举字段定义完整
- 🟡 设计层面：字节序、校验算法、量程边界、ACK/重试策略、长时间运行的回卷、状态机切换边界、单位标注完整性
- 🟢 小问题：错别字、空标题占位、括号嵌套、单位大小写、章节内引用失效

**不要做**：
- 不要拉 UI 文档
- 不要评论"UI 上是否有对应入口"
- 不要评论"按钮名是否对齐"

### 模式 B：审「上位机界面设计文档」

**触发**：用户说「检查界面设计文档」「审 UI 文档」「检查上位机文档」，或给出界面设计文档的 URL/路径。

**范围**：审 UI 文档本身，**同时**主动拉取该文档内链接 / 引用的通信协议文档做交叉对账。如果 UI 文档里没有显式链接，先问用户拿协议文档位置，没拿到不要硬猜。

**第一步是发现并拉取协议文档**：
1. 先 fetch UI 文档
2. 在 fetch 结果里搜协议文档的引用（典型形式：飞书 docx 链接、章节里写的"详见《xxx 通信协议》"、内嵌的 docx token）
3. 找到后用 `lark-cli docs +fetch` 把协议文档也拉下来作为对账底稿
4. 找不到就跟用户要

**跑这些检查项**：
- 🔴 跨文档一致性：UI 输入项数量 = 协议字段数量、上行反馈字段 = UI 显示项、协议定义的帧在 UI 是否有触发入口（死帧）、位图字段的 UI 展示方式
- 🔴 顺带核一遍协议侧的 DATA 长度和功能码（因为 UI 错配往往是协议改了引起的，不查协议判不准谁对谁错）
- 🟡 跨文档命名同步、ACK/重试策略一致、量程边界 UI 是否做了校验、启停 vs 参数设定的联动语义
- 🟡 UI 文档自身：状态机切换的边界态、回卷处理的 UI 表现
- 🟢 小问题：UI 文档的错别字、占位、命名、跨文档引用失效

**输出时要标清楚每条问题来源**：是 UI 文档独有问题，还是协议-UI 对不上的问题（对不上的要同时给出两边各自的位置）。

## 通用工作流

1. **确认模式**（A 还是 B）和资源位置（飞书 docx token / URL / 本地路径），是否含白板。
2. **全文拉取** ——
   - 飞书 docx：`lark-cli docs +fetch --doc <token> --format pretty`
   - 飞书白板：`lark-cli docs +media-download --type whiteboard --token <token> --output ./xxx.png --overwrite`，然后用 Read 看图
   - 本地：直接 Read
   - 模式 B 还要额外拉协议文档
3. **逐项对账**，按严重程度 🔴/🟡/🟢 分级。
4. **不要默认改文档** —— 列出问题让用户拍板；用户说"修"再用 `docs +update --mode replace_range --selection-by-title` 改。
5. **修改后必须重新 fetch 验证**，不要相信 `success: true` 就完事。

## 检查清单（按模式选用，见上方"工作模式"）

### 🔴 协议自洽（模式 A、B 都查）

#### 1. DATA 长度对账
协议帧表里的"数据长度"字段必须等于数据域表中所有字段字节数之和。
- 公式：u8=1, u16=2, u32=4, float=4, double=8
- 每次数据域增删字段都要重算 DATA 长度
- **最高频踩坑点**：开发者在数据域加 / 删一个字段，忘了同步改 DATA 长度 → 解析错位、CRC 位置错、整条帧报废

#### 2. 功能码一致性（白板 / 时序图 vs 协议章节）
如果文档里有用白板 / 时序图画通信流程，每个流程标注的功能码必须和协议章节的功能码定义一致。
- 常见踩坑：画图时复制粘贴上一个流程框忘改功能码
- 不仅看功能码数字，**还要看流程框里的文字说明**是否对应

#### 3. 位图 / 枚举字段定义完整
错误码、状态字这类位图字段必须给出每个 bit 的含义对照表，不能只写"详见后续补充"。

### 🔴 跨文档一致性（仅模式 B）

#### 4. UI 输入项数量 = 协议字段数量
下行参数帧（上位机 → 下位机）有几个字段，UI 上就必须有几个输入框 / 控件。
- 缺一个 = 那个字节没数据来源，要么硬编、要么发非法值
- 多一个 = UI 上有但协议没承载，用户改了没生效

#### 5. 上行反馈字段 = UI 显示项
传感器 / 状态上报帧的每个字段都要在 UI 上有显示位置。否则数据收了没用。

#### 6. 协议定义了但 UI 没触发入口的"死帧"
检查每个功能码在 UI 文档里是否能找到触发路径（按钮、自动时机、定时心跳）。

#### 7. 位图字段的 UI 展示方式
- 多 bit 可同时置位 → UI 必须支持同时展示多个故障
- bit 含义对照表应在 UI 弹窗中直接列出，而不是"参考协议文档"跨文档跳转

### 🟡 设计层面

#### 8. 字节序约定（模式 A、B）
小端 / 大端必须在协议文档明确写。模式 B 还要检查 UI 解析假设是否一致。

#### 9. 校验算法明确（模式 A、B）
CRC8 / 16 / 32 必须写明 polynomial / init / reflect / xorout（例如 "CRC-8/MAXIM"、"CRC-16/MODBUS"）。

#### 10. 单位标注完整性（模式 A 查协议、模式 B 查两边）
每个数值字段都要标单位：
- 转速 rps/rpm/Hz、功率 W（大写）、电压 V/mV、电流 A/mA/0.1A
- 温度 °C/0.1°C、压力 MPa/kPa、时间 ms/μs、占空比 %/0.1%
- 重量 / 位置原始值必须给量程或换算系数
- 时间戳：u32 约 49.7 天回卷、u16 约 65 秒回卷

#### 11. 量程 / 范围边界
- 模式 A：协议字段的合法范围（如 `[30,90]`）要在协议里明确
- 模式 B：还要检查 UI 输入有无范围校验、UI 文档能否查到该范围

#### 12. ACK / 重试策略
- 模式 A：协议明确"几次无应答 = 异常"
- 模式 B：UI 的 ACK 异常弹窗逻辑必须和协议侧重试次数对齐

#### 13. 跨文档命名同步（仅模式 B）
按钮名 ↔ UI 文档描述 ↔ 协议文档功能描述 ↔ 白板按钮标签全部对齐。命名要避免误导（例如"清空错误码"是清当前故障还是清计数器）。

#### 14. 启停 vs 参数设定的联动语义（仅模式 B）
- 启动顺序：先设参数还是先启动？
- 运行中改参数是否立即生效？
- 停机后设定值是否保留？

#### 15. 长时间运行的回卷 / 累积误差
- u32 时间戳 49.7 天回卷
- 累积计数器溢出后图表跳变需 wrap-around 处理

#### 16. 状态机切换的边界态（侧重 UI，模式 B）
- 数据反馈态 ↔ 不反馈态切换时，显示数据如何处理（保留 / 清空 / 置灰）？
- 连接断开 / 恢复时，状态显示如何重置？

### 🟢 小问题

- **错别字 / 重字**：常踩"的的"、"在不"、"是是"
- **空标题占位**：标题后无内容，补"等待补充"或删除
- **括号嵌套**：例如"脉冲（pulses）"、"开度（%开度）"
- **大小写**：物理单位 W / V / A 必须大写
- **链接 / 引用失效**：跨文档引用章节，章节名改了 / 删了

## 输出格式

```markdown
## 审核范围
- 模式：A（仅协议）/ B（UI + 关联协议）
- 文档：<文档名 + token/路径>
- （模式 B）关联协议文档：<文档名 + token/路径>

## 🔴 严重
1. **<问题>** — <来源：协议 / UI / 跨文档> — <位置：章节标题 / 行号 / 白板区块> — <建议修法>

## 🟡 设计层面
...

## 🟢 小问题
...

## ✅ 已对齐
- ...
```

每条问题必须给出**具体位置**和**来源标记**（协议 / UI / 跨文档），避免泛泛而谈。

## 修文档时的注意事项

- **整段重写最稳**：`lark-cli docs +update --mode replace_range --selection-by-title "<完整标题>"`
- **修改前先 fetch 底稿**：避免覆盖用户的其他改动
- **白板内容无法用 lark-doc 编辑**：只能让用户在飞书里手动改，或走 lark-whiteboard 全量覆盖（破坏性）
- **v1 API 有 `[deprecated]` 警告但仍可用**；如果用 v2，用 `--api-version v2 --command str_replace --pattern <regex>`
- **传文件路径要用相对路径**：`@./xxx.md`，不能 `@/tmp/xxx.md`
- **修改后必须重新 fetch 验证**

## 常见踩坑（来自历史经验）

1. **数据域字段增删时忘改 DATA 长度** —— 最高频，每次必查（A、B 都查）
2. **白板复制流程框忘改功能码** —— 高频（A、B 都查）
3. **拆字段后两个字段对齐错位** —— 比如把单字段 u16 拆成两个 u16 后，DATA 长度只加了 2 没加 4
4. **协议改了但 UI 文档没同步** —— 按钮名、字段数、范围都可能漂（仅 B 能发现）
5. **白板里的描述文字和功能码不匹配** —— 功能码改对了但说明文字还是旧的
6. **"清空 X" 类按钮的语义** —— 究竟清的是当前值还是计数器要写清楚（仅 B）
7. **位图字段当数值显示** —— UI 把 u16 位图当成枚举或十进制数显示（仅 B）
8. **死帧** —— 协议定义了某个帧但 UI 没任何触发入口（仅 B）
