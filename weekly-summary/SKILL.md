# weekly-summary

从 Claude Code 的每日会话记录或已生成的日报中，提炼一份结构化的每周工作总结。

## 核心定位 vs daily-summary

- **daily-summary**：面向单日，还原一天内每个会话的细粒度信息（用户原话、打断点、半自动信号等）
- **weekly-summary**：面向一周，**提炼主线而非罗列细节**。强调项目级进展、阶段跃迁、里程碑密度、闭环率，服务于周报/复盘而不是事故回放

一句话差异：日报回答「今天发生了什么」，周报回答「这一周我们把哪些项目往前推了多远」。

## 数据源（双源融合 + 兜底）

两个源**定位互补、不是二选一**：飞书汇报是用户手写的浓缩主线（每个工作日齐全、字段结构化，但一天压成几百字、没有 commit/数值等细节）；本地 daily-summary 是会话自动还原的细节底稿（每天数千字，含提交号、bug 根因、技术决策，但只覆盖跑过的天）。**汇报定主线和闭环结论，本地日报补里层细节和未闭环尾巴。**

> **采集已固化为 [`assets/collect.py`](assets/collect.py)** —— 一条命令跑完「现场取本人 open_id → 拉区间汇报 → 客户端按本人过滤 → 每天配本地/邮箱细节」，按周一~周六输出 JSON digest（周日不计）。**优先直接调用它**，下面各源的底层命令与坑位仅供理解和排错：
>
> ```bash
> python3 assets/collect.py --pretty                 # 默认本周（今天所在 ISO 周的周一起）
> python3 assets/collect.py --monday 2026-05-25       # 指定某周（传该周周一）
> ```
>
> 输出顶层：`week`（周一/周六）、`me`（本人 open_id）、`stats`（report_days / detail_local / detail_mail / detail_none / weekly_reports）、`days[]`、`weekly_reports[]`。每个 `days[i]`：`date`·`weekday`·`report`（`{rule, fields:[{name,value}]}` 或 null）·`detail`（`{source: local|mail|none, text}`）。拿到 digest 后直接进「第二步：按项目聚合」。脚本是 fail-first 的：lark-cli 缺失 / 未登录 / 飞书报错都会抛出，不静默兜底。

### 数据源1（主线，权威）：飞书「汇报」里本人手写的日报/周报

这是用户**亲手写、主观筛过**的内容，作为周报骨架、闭环判定和「下周计划」的第一依据。用 user 身份调汇报查询接口 `POST /open-apis/report/v1/tasks/query`。

三个必须遵守的坑（实测确认）：
1. **必须 `--as user`**：user_access_token 直接能读本人有权查看的汇报，**无需给应用加 tenant 级 scope**（用 `--as bot` 会因缺 `report:task:readonly` 报 99991672）。
2. **服务端 `user_id` 过滤无效**：传了也会返回全组所有人的汇报，**必须在客户端按 `from_user_id == 本人 open_id` 过滤**。
3. **open_id 不硬编码**：每次现场取（遵守全局敏感信息规则）。

```bash
# 现场取本人 open_id（绝不硬编码）。注意：裸调返回的 data 直接是用户字段
# （data.open_id，无 data.user 这层），且 lark-cli 的 -q jq 对裸调 api 取不到值，
# 所以用 python 解析（collect.py 已封装这一步）
ME=$(lark-cli api GET /open-apis/authen/v1/user_info --as user \
  | python3 -c 'import json,sys; print(json.load(sys.stdin)["data"]["open_id"])')
# 本周时间窗（epoch 秒）：周一 00:00 到周日 00:00（不含）→ 实际只覆盖周一~周六；按实际区间替换日期
# 例：本周一 2026-05-25，则 END 取本周日 2026-05-31 00:00，统计止于周六 05-30 23:59:59
START=$(date -d '2026-05-25 00:00' +%s); END=$(date -d '2026-05-31 00:00' +%s)
lark-cli api POST /open-apis/report/v1/tasks/query --as user \
  --data "{\"commit_start_time\":$START,\"commit_end_time\":$END,\"page_token\":\"\",\"page_size\":20}"
```

返回 `data.items[]`，每条是一份汇报：
- 客户端先 `filter(from_user_id == $ME)`，再按 `rule_name` 区分「工作日报」/「工作周报」
- 正文在 `form_contents[]`，每项 `field_name`（如「今日总结」「明日计划」；周报为「本周关键工作清单/工作总结/分享/下周关键工作计划」）+ `field_value`
- **`field_value` 内无换行**：一天多个项目被压成一整行，靠「`项目名（…，全天主力）：`」这种文字标记切块（按 `）：` 或项目名模式切分），拿不到原始段落
- 条数多时分页：`has_more` 为真则带上返回的 `page_token` 续拉

### 数据源2（细节补充）：本地 daily-summary，缺失则飞书邮箱回退

对数据源1 覆盖到的每一天，补一份细节底稿。**先查本地，本地没有再查邮箱。**

```bash
# a) 本地：定时任务产物固定在 ~/.local/state/daily-summary/，手动跑的可能在工作目录/桌面
find ~/.local/state/daily-summary -maxdepth 1 -name "daily-summary-*.html" -newermt "7 days ago" 2>/dev/null
find ~/Desktop . -maxdepth 3 -name "daily-summary-*.html" -newermt "7 days ago" 2>/dev/null
```

```bash
# b) 本地缺某天 → 从飞书邮箱取那天的「每日工作总结」邮件（同一天常有多封重复，取时间最新那封）
lark-cli mail +triage --as user --query "每日工作总结 — YYYY-MM-DD"   # 拿 message_id
lark-cli mail +message --as user --message-id <message_id>           # 取正文 HTML
```

从 HTML 提炼时直接读正文文本即可（标签忽略）；日报已做过「跳过 system-reminder / 识别 skill 调用 / 按项目归类」这些重活，周报只做二次提炼。某天本地和邮箱都没有，就标注该天「仅汇报主线、无细节源」，不要臆造细节。

### 兜底源：jsonl 原始会话
仅当**汇报和本地/邮箱日报都缺**（例如用户既没用飞书汇报、也没跑过 daily-summary），或用户明确要求从原始会话重算时，才扫描 `~/.claude/projects`：
```bash
find ~/.claude/projects -name "*.jsonl" -newermt "7 days ago" -printf "%T+ %s %p\n" | sort -r
```
抽消息方法和 daily-summary 完全一致（见 daily-summary SKILL.md 第二步），差异只在时间过滤用一周（周一~周六）窗口而不是 1 天。

### 时间窗口定义
**统计口径：周一到周六，周日不计入**（即每周 6 个工作日，周日休息不纳入周报）。
- **默认**：本周的周一 00:00 到周六 23:59:59（= 周日 00:00 之前）。即使在周日跑 `/weekly-summary`，也只统计本周一~周六，当天（周日）不计
- **指定 ISO 周**：如果用户说「上周」或指定周号（如 `2026-W17`），取该 ISO 周的周一到周六，**丢弃周日**
- **自定义区间**：支持 `weekly-summary 2026-04-15 2026-04-22` 这种起止日期；若区间内含周日，同样把周日剔除再统计
- **实现**：时间窗按周一 00:00 到周日 00:00（不含）取，落在周日的汇报/日报/会话一律过滤掉

## 流程

### 第一步：确定区间 + 采集（跑 collect.py）
1. 解析用户输入的时间窗口，没说就用本周（周一~周六，周日不计）
2. **跑采集脚本**：`python3 assets/collect.py --pretty`（指定周则加 `--monday <该周周一>`）。它已把数据源1（飞书汇报）+ 数据源2（本地 HTML，缺则邮箱回退）一次跑完，返回按日 digest
3. **看 `stats` 给用户对账**：`report_days` 覆盖了几天、细节有几天来自本地 / 几天邮箱回退 / 几天缺（`detail_none`）；缺细节的日期只用汇报主线，不臆造
4. **兜底判断**：若 digest 里 `report_days` 和有细节的天都很少（用户既没用飞书汇报、也没跑过 daily-summary），才回退到 jsonl 抽取（见兜底源）

### 第二步：按项目聚合
**先以数据源1（汇报）定主线**：每个项目本周做了什么、推到什么状态、闭环了什么、下周计划，以用户手写的汇报为准。**再用数据源2（本地/邮箱日报）补细节**：填充该项目的 commit hash、PR 号、release tag、bug 根因、关键数值等汇报里被压掉的内容。汇报和日报冲突时以汇报为准（那是用户的主观结论），但日报里出现、汇报漏掉的重要交付要补进来并可在时间线标注。

对每个项目（以 `cwd` 或日报里的项目名为 key）合并整周信息：

- **项目主题**：用最高频出现的技术关键词总结（不是每天重复抄）
- **阶段跃迁**：本周从 X 推到 Y（例如「需求讨论 → 架构审核 → dev-execute 中段」），标记穿越了哪些 skill 里程碑
- **里程碑计数**：本周该项目共 N commit / M PR / K release
- **持续时长**：本周投入总时长（各日时段相加，注明真实工作时长而非跨度）
- **状态变化**：周一状态 → 周六状态（已合并 / WIP / 卡住 / 搁置）
- **关键决策**：本周在这个项目上做过哪些有记忆价值的取舍（只挑 1-3 条最重要的，不要把每天的决策都搬过来）

### 第三步：识别跨日主线
日报里的「跨项目共识」是单日内多项目的交集；周报需要额外识别**跨日主线**：

- **反复出现的技术话题**：例如「相机管理」「终端持久化」「网络路由排查」贯穿多日 —— 判定为本周的深耕主题
- **从提出到闭环的完整链路**：有哪些问题是周一提出、周中推进、周末闭环的（最有成就感的部分）
- **悬而未决的尾巴**：哪些问题周一就出现、到周六（一周结束）还没解决（最值得优先处理的部分）
- **上下文切换成本**：统计本周同时推进的项目数（>3 个活跃项目通常意味着精力分散）

### 第四步：组织内容并生成 HTML（三层结构）

**不生成中间 md 文件**，直接产出 HTML。顶层摘要可直接粘到周报；中层项目进展供 1-on-1 或月度复盘引用；底层时间线/未闭环供本人内省。下面的 markdown 仅说明内容结构，实际落盘的是按此结构渲染的 HTML（见第五步）：

```markdown
# 每周工作总结 — YYYY-MM-DD 至 YYYY-MM-DD（第 X 周）

## 摘要（可提交）

本周主攻 N 个项目，完成 X commit / Y PR / Z release，闭环主要问题 A / B / C；仍在推进 D / E。

- **项目A**（主攻）：本周推进主线（阶段 → 阶段）；关键里程碑（X commit + Y PR + 发版 vX.Y.Z）
- **项目B**：本周主要做了什么、阶段跃迁
- **项目C**：本周主要做了什么
- **零散/辅助**：一句话带过（环境配置、调研、文档）
- **本周主线**（可选，1-2 句）：跨项目的技术共识或主题

---

## 项目进展（按投入量排序）

### 项目A
- **本周主线**：一句话概括本周在这个项目上推进的核心目标（不是每天做了什么，而是「把 X 从 A 状态推到 B 状态」）
- **阶段跃迁**：周一处于 X → 周六处于 Y
- **投入时长**：~Xh（分布在 N 天，真实工作时长）
- **里程碑**：
  - YYYY-MM-DD: commit <sha> - <一句话>
  - YYYY-MM-DD: PR #N <标题> merged / open
  - YYYY-MM-DD: release vX.Y.Z
- **关键决策**（≤3 条）：
  - 决策内容 + 为什么这么定（一句话）
- **闭环问题**：
  - 周X 提出「…」→ 周Y 解决
- **未闭环问题**：
  - 周X 提出「…」仍未解决，当前卡在 <具体点>
- **下周计划**：具体到 PR / 文件 / 函数，不要写「继续推进」

### 项目B
（同上结构）

---

## 本周时间线（里程碑密度视图）
按日期列出每天的 2-3 条最关键事件，便于回顾节奏：

- **周一 YYYY-MM-DD**：项目A 立项；项目B 起 PR
- **周二 YYYY-MM-DD**：项目A 架构审核通过；项目C 调研 BGP 路由
- ...
- **周六 YYYY-MM-DD**：项目A 合并；本周收尾

## 闭环与未闭环汇总

### 本周闭环（可以放下的）
- 项目A：X 问题（周一提出 → 周三解决）
- 项目B：Y 问题（周二提出 → 周六合并）

### 未闭环（优先下周处理）
- 项目A：X 问题，卡在 <具体点>，需 <具体行动>
- 项目B：Y 问题，已 WIP 3 天，下周先做 <具体事>
- 项目C：调研悬空，需拍板 <决策>

## 本周观察（选填）
- **上下文切换**：本周同时活跃 N 个项目，主要切换日是周X / 周Y
- **里程碑分布**：commit 集中在周几、PR 集中在周几（反映交付节奏）
- **情绪 / 节奏信号**：本周有没有密集打断、大段空白、深夜工作等异常节奏
- **跨项目主线**：本周有 M 个项目共享同一技术话题 / 架构决策
```

### 输出约束

- **摘要部分**：≤5 条项目 bullet + 1 条可选主线，直接粘到周报即可；顶部加一句话总量统计（commit/PR/release 数、闭环/未闭环数）
- **项目进展**：按本周真实投入时长降序；每个项目必须有「阶段跃迁」和「下周计划」，否则删掉这个项目只留一行摘要
- **时间线**：每天 2-3 条即可，追求节奏感而不是完整性；没有事件的日子直接跳过或标「idle」
- **闭环 vs 未闭环**：这两部分是周报最有价值的部分。闭环给自己正反馈，未闭环给下周定优先级

### 第五步：落盘 HTML 并发送邮件

把第四步组织好的内容直接渲染为 HTML 落盘，再发到本人飞书邮箱。**全程不产出 md 文件。**

> 邮件地址等敏感信息**不要写进本文档或任何产物**，发送时用 lark-cli 现场获取，绝不硬编码。

- **HTML 渲染**：参考 `../daily-summary/assets/template.html` 的样式骨架（内联样式、标题/列表/分割线、`<code>` 标识符），按本 skill 的三层结构填充。HTML 落盘为 `weekly-summary-YYYY-Www.html`（如 `weekly-summary-2026-W17.html`），路径默认当前工作目录，除非用户指定别处。
  - **禁止 `<button>` / `<script>` / `onclick` 等任何依赖 JS 的交互元素**：邮件客户端会剥离 JS，按钮点了无效，只会留个死控件。
- **获取本人邮箱地址**（不要硬编码，每次现场查）：

  ```bash
  ME=$(lark-cli mail user_mailboxes profile --params '{"user_mailbox_id":"me"}' -q '.data.primary_email_address')
  ```

- **发送邮件**：

  ```bash
  lark-cli mail +send \
    --to "$ME" \
    --subject "每周工作总结 — YYYY-Www" \
    --body "$(cat weekly-summary-YYYY-Www.html)" \
    --confirm-send
  ```

  - body 是 HTML，lark-cli 自动识别为 HTML 邮件，不要加 `--plain-text`
  - `--confirm-send` 直接发送；去掉则只存草稿。默认直接发，首次或用户要求确认时先存草稿让用户过目
  - 发送前确认 token 未过期（`lark-cli doctor`），过期会发送失败，需重新授权
- 发送成功后报告：HTML 路径、邮件已发送（地址来自 lark-cli，可不在对话里回显完整邮箱）

## 与 daily-summary / 飞书汇报的配合建议

- 理想流程：每天在飞书「汇报」里写工作日报（主线源），同时每天下班前跑一次 `/daily-summary`（细节源），周六收尾时（或周日补跑，统计仍只到周六）跑 `/weekly-summary` 即可双源融合
- 飞书汇报是手写浓缩版（约几百字/天），本地 daily-summary 是会话还原（约数千字/天）——前者权威定调、后者补细节，差约一个数量级，所以要融合而非择一
- 若本周本地日报没跑（缺 HTML），优先从飞书邮箱回退取；连邮箱也没有，该天就只用汇报主线
- 若本周连飞书汇报都没写、本地也没跑，才回退扫描 jsonl，耗时更长且细节降级
- 日报关注「原话 / 打断 / 纠正」，周报关注「阶段 / 里程碑 / 闭环」—— 不要在周报里把每天的打断信号都搬过来，那是日报干的活

## 注意事项

- **不要抄日报**：周报是二次提炼。看到日报里的 5 条子弹不要原样搬进来，而是合并成一句主线
- **不要臆造里程碑**：只写 jsonl / 日报里真实出现过的 commit hash / PR 号 / release tag。没抽到就说「本周无发版」，不要填充
- **尊重用户写作风格**：如果 memory 里有用户的写作风格偏好（如「简洁直白、技术细节充分、口语化」），按那个风格写
- **克制篇幅**：周报比日报更长，但每个项目的描述应比 6 份日报的对应段落加起来短 —— 提炼而不是堆砌
- **标注数据源**：文末加一行注明本次用了哪些源，便于事后追溯，例如「数据源：飞书汇报 X 篇（主线）+ 本地 daily-summary Y 份 + 邮箱回退 Z 份 + jsonl W 个会话」；没用到的源省略
- **以汇报为主线、不被细节淹没**：本地日报字数是汇报的约 10 倍，融合时别把日报的细节原样堆进周报——汇报怎么定调，周报就怎么定调，日报只在对应项目下补 1-3 条关键细节（提交号/数值/根因）
- **跳过当前会话本身**：生成周报的那个会话通常信息量有限，可不单列
