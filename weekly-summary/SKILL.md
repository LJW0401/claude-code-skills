# weekly-summary

从 Claude Code 的每日会话记录或已生成的日报中，提炼一份结构化的每周工作总结。

## 核心定位 vs daily-summary

- **daily-summary**：面向单日，还原一天内每个会话的细粒度信息（用户原话、打断点、半自动信号等）
- **weekly-summary**：面向一周，**提炼主线而非罗列细节**。强调项目级进展、阶段跃迁、里程碑密度、闭环率，服务于周报/复盘而不是事故回放

一句话差异：日报回答「今天发生了什么」，周报回答「这一周我们把哪些项目往前推了多远」。

## 数据源（优先级）

### 优先源：已有的 daily-summary 文件
周总结应优先扫描最近 7 天的 `daily-summary-YYYY-MM-DD.md`（通常在用户工作目录下或 `~/Desktop/feishu/` 之类固定位置），用它们做聚合：
```bash
# 常见位置，按用户习惯调整
find ~/Desktop -maxdepth 3 -name "daily-summary-*.md" -newermt "7 days ago" 2>/dev/null
find . -maxdepth 2 -name "daily-summary-*.md" -newermt "7 days ago" 2>/dev/null
```
日报已做过「跳过 system-reminder / 识别 skill 调用 / 按项目归类」这些重活，周总结只需要做二次提炼。

### 回退源：jsonl 原始会话
如果找不到足够的日报文件（<5 天），或用户明确要求从原始会话重算，则扫描 `~/.claude/projects`：
```bash
find ~/.claude/projects -name "*.jsonl" -newermt "7 days ago" -printf "%T+ %s %p\n" | sort -r
```
抽消息方法和 daily-summary 完全一致（见 daily-summary SKILL.md 第二步），差异只在时间过滤用 7 天窗口而不是 1 天。

### 时间窗口定义
- **默认**：最近 7 个自然日（含今天），即 `today - 6 days` 到 `today`
- **指定 ISO 周**：如果用户说「上周」或指定周号（如 `2026-W17`），按 ISO 8601 周一到周日
- **自定义区间**：支持 `weekly-summary 2026-04-15 2026-04-22` 这种起止日期

## 流程

### 第一步：确定区间 + 定位输入
1. 解析用户输入的时间窗口，没说就用最近 7 天
2. 列出区间内所有 `daily-summary-*.md`，输出清单给用户确认覆盖哪些日期
3. 若缺失某天日报（例如周末没工作、或用户没跑过 daily-summary），标注「该日无日报，已回退到 jsonl 抽取」并对缺失日期跑一次轻量 jsonl 抽取作为补丁

### 第二步：按项目聚合
对每个项目（以 `cwd` 或日报里的项目名为 key）合并整周信息：

- **项目主题**：用最高频出现的技术关键词总结（不是每天重复抄）
- **阶段跃迁**：本周从 X 推到 Y（例如「需求讨论 → 架构审核 → dev-execute 中段」），标记穿越了哪些 skill 里程碑
- **里程碑计数**：本周该项目共 N commit / M PR / K release
- **持续时长**：本周投入总时长（各日时段相加，注明真实工作时长而非跨度）
- **状态变化**：周一状态 → 周日状态（已合并 / WIP / 卡住 / 搁置）
- **关键决策**：本周在这个项目上做过哪些有记忆价值的取舍（只挑 1-3 条最重要的，不要把每天的决策都搬过来）

### 第三步：识别跨日主线
日报里的「跨项目共识」是单日内多项目的交集；周报需要额外识别**跨日主线**：

- **反复出现的技术话题**：例如「相机管理」「终端持久化」「网络路由排查」贯穿多日 —— 判定为本周的深耕主题
- **从提出到闭环的完整链路**：有哪些问题是周一提出、周中推进、周末闭环的（最有成就感的部分）
- **悬而未决的尾巴**：哪些问题周一就出现、到周日还没解决（最值得优先处理的部分）
- **上下文切换成本**：统计本周同时推进的项目数（>3 个活跃项目通常意味着精力分散）

### 第四步：输出 md（三层结构）

顶层摘要可直接粘到周报；中层项目进展供 1-on-1 或月度复盘引用；底层时间线/未闭环供本人内省。

```markdown
# 每周工作总结 — YYYY-MM-DD 至 YYYY-MM-DD（第 X 周）

## 摘要（可提交）

本周主攻 N 个项目，完成 X commit / Y PR / Z release，闭环主要问题 A / B / C；仍在推进 D / E。

- **项目A**（主攻）：本周推进主线（阶段 → 阶段）；关键里程碑（X commit + Y PR + 发版 vX.Y.Z）
- **项目B**：本周主要做了什么、阶段跃迁
- **项目C**：本周主要做了什么
- **零散/辅助**：一句话带过（环境配置、调研、文档）
- **本周主线**（可选，1-2 句）：跨项目的技术共识或主题

---

## 项目进展（按投入量排序）

### 项目A
- **本周主线**：一句话概括本周在这个项目上推进的核心目标（不是每天做了什么，而是「把 X 从 A 状态推到 B 状态」）
- **阶段跃迁**：周一处于 X → 周日处于 Y
- **投入时长**：~Xh（分布在 N 天，真实工作时长）
- **里程碑**：
  - YYYY-MM-DD: commit <sha> - <一句话>
  - YYYY-MM-DD: PR #N <标题> merged / open
  - YYYY-MM-DD: release vX.Y.Z
- **关键决策**（≤3 条）：
  - 决策内容 + 为什么这么定（一句话）
- **闭环问题**：
  - 周X 提出「…」→ 周Y 解决
- **未闭环问题**：
  - 周X 提出「…」仍未解决，当前卡在 <具体点>
- **下周计划**：具体到 PR / 文件 / 函数，不要写「继续推进」

### 项目B
（同上结构）

---

## 本周时间线（里程碑密度视图）
按日期列出每天的 2-3 条最关键事件，便于回顾节奏：

- **周一 YYYY-MM-DD**：项目A 立项；项目B 起 PR
- **周二 YYYY-MM-DD**：项目A 架构审核通过；项目C 调研 BGP 路由
- ...
- **周日 YYYY-MM-DD**：项目A 合并；整理周报

## 闭环与未闭环汇总

### 本周闭环（可以放下的）
- 项目A：X 问题（周一提出 → 周三解决）
- 项目B：Y 问题（周二提出 → 周日合并）

### 未闭环（优先下周处理）
- 项目A：X 问题，卡在 <具体点>，需 <具体行动>
- 项目B：Y 问题，已 WIP 3 天，下周先做 <具体事>
- 项目C：调研悬空，需拍板 <决策>

## 本周观察（选填）
- **上下文切换**：本周同时活跃 N 个项目，主要切换日是周X / 周Y
- **里程碑分布**：commit 集中在周几、PR 集中在周几（反映交付节奏）
- **情绪 / 节奏信号**：本周有没有密集打断、大段空白、深夜工作等异常节奏
- **跨项目主线**：本周有 M 个项目共享同一技术话题 / 架构决策
```

### 输出约束

- **摘要部分**：≤5 条项目 bullet + 1 条可选主线，直接粘到周报即可；顶部加一句话总量统计（commit/PR/release 数、闭环/未闭环数）
- **项目进展**：按本周真实投入时长降序；每个项目必须有「阶段跃迁」和「下周计划」，否则删掉这个项目只留一行摘要
- **时间线**：每天 2-3 条即可，追求节奏感而不是完整性；没有事件的日子直接跳过或标「idle」
- **闭环 vs 未闭环**：这两部分是周报最有价值的部分。闭环给自己正反馈，未闭环给下周定优先级
- **输出路径**：默认当前工作目录下的 `weekly-summary-YYYY-Www.md`（如 `weekly-summary-2026-W17.md`），除非用户指定别处

## 与 daily-summary 的配合建议

- 理想流程：每天下班前跑一次 `/daily-summary`，周五（或周日）跑一次 `/weekly-summary`，周总结能直接复用 7 份日报
- 若本周没跑过日报，周总结仍然可用（回退扫描 jsonl），但耗时更长且细节会降级
- 日报关注「原话 / 打断 / 纠正」，周报关注「阶段 / 里程碑 / 闭环」—— 不要在周报里把每天的打断信号都搬过来，那是日报干的活

## 注意事项

- **不要抄日报**：周报是二次提炼。看到日报里的 5 条子弹不要原样搬进来，而是合并成一句主线
- **不要臆造里程碑**：只写 jsonl / 日报里真实出现过的 commit hash / PR 号 / release tag。没抽到就说「本周无发版」，不要填充
- **尊重用户写作风格**：如果 memory 里有用户的写作风格偏好（如「简洁直白、技术细节充分、口语化」），按那个风格写
- **克制篇幅**：周报比日报更长，但每个项目的描述应比 7 份日报的对应段落加起来短 —— 提炼而不是堆砌
- **标注数据源**：文末加一行「数据源：X 份日报 + Y 个 jsonl 会话」，便于事后追溯
- **跳过当前会话本身**：生成周报的那个会话通常信息量有限，可不单列
